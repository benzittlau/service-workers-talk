One of the major roles modern javascript frameworks serve is to allow you to combine data with a template to create a DOM structure. There was a time when it might have been useful simply to do this once, but modern javascript apps often have changing data and state, and it is necessary to make sure the interface reflects any changes to that data. Change detection is the first part of that process; allowing the framework to be aware that changes have occurred, so they can handle those changes appropriately.

So why should we care about change detection?

DOM changes are slow, so it's hard to make an app fast

Different approaches used by the different frameworks

No "right" answer

I personally find the different ways in which frameworks handle the challenges of change detection of DOM mutation incredibly interesting. Making a web application "feel" fast is very hard (as anyone who's attempted to make an interactive application without the modern tools can likely attest). Each framework has taken a different approach, and neither of these approaches has "won". As the frameworks are converging towards similar concepts in terms of components and data and action flows, their change detection and update approaches have remained very different. How the framework handles these questions has direct impact on how the user of the framework has to work, so understanding these differences gives a better appreciation for the usage differences between the frameworks.

        return this.people

Emberisms (Computed)



export default Component.extend({
  ...
  filteredPeople: Ember.computed('filter', 'people', function() {
    const people = Ember.get(this, 'people');
    const filter = Ember.get(this, 'filter');

    if (filter === 'all') {
      return people
    }

    var that = this;
    return people.filter(function(person) {
      return person.role === that.filter
    })
  }),

Some Thoughts

  • Really neat features inspired from other frameworks
  • Lots of other great features I didn't have time to show
  • Accomplishes it's goals (accessible and progressive) really well
  • Comparable performance to other frameworks
  • Excellent documentation

But...

Good for...

  • Replacing jQuery as a tool to add behaviour to an existing DOM
  • Projects with casual contribution you want to lower the barrier to entry on

Not sure about...

  • Using for full SPA applications vs. the other alternatives
    • Pretty immature testing support
    • Has some strong core projects (router, cli), but less mature community options
    • No strong "Tier 1" backers (largely a one man show)
    • No strong reference projects in production (GitLab)
At the end of the day I liked it a lot, and could see it becoming an easy to use "go to" for side projects
